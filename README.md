# ssbu-anti-jp-error

スマブラSP の「アヴァるエラー」対策の Skyline プラグイン．  
対戦相手が意図的なエラーを押し付けてくる攻撃を検知し，無効化します．  
アヴァるエラーについて，詳しくは[アヴァるエラーとは](##アヴァるエラーとは)とはを参照してください．

## 機能

マッチングの際に特定のレスポンスが返ってきたプレイヤーをチーターと判定し，該当するレスポンスを書き換えて「自分がホストの空セッション」に偽装します．  
その結果，こちらは相手と通信せず，エラーを出さずにトレモへ進み，自動で次のプレイヤーを検索します．ペナルティ等は発生しません．  
またブロックしたプレイヤーを学習し，次からはIDが判明した瞬間に自動でブロックします．

### 検知

以下のどれかを満たせばチーターとみなす:

- `application_buffer[0] == 0x02` (アヴァる系が付ける固定マーカー)
- `max_participants > 2` (1v1にも関わらず3人以上を集めるセッション)
- 過去に検知した PID (ブロックリスト)

検知時は画面上に短いテロップが出て，相手の名前を表示し，数秒後にで自動で閉じます．

## 再検索

検知後，次のマッチ成立までが**30秒～2分程度**と少し長いです．  
これはサーバー側の再通知間隔が律速で，クライアント側からこれ以上は縮められない可能性があり，改善のため開発中です (公開されているものにはこの機能がないため，プロトタイプとしています)．

## ダウンロード

[リリースページ](https://github.com/Atamol/ssbu-anti-jp-error/releases/tag/v0.1.0)からダウンロードできます．

## 動作環境

- バージョン 13.0.4
- Atmosphère + Skyline が動くCFW環境

## 注意

- 再検索の項目にもある通り，次のマッチまでの間が長いですが，現時点では正常です
- 判定は保守的に絞ってますが，誤検知が完全にゼロとは言い切れません (エラー流しの犯人は100%ブロックできますが，それ以外も検知している可能性がないとは言い切れません)
- まだプロトタイプの段階です．想定外の挙動があればIssueを送るか，何らかでDMをください

# アヴァるエラーとは

**アヴァるエラー** (別名: エスケーエラー) は，大乱闘スマッシュブラザーズ SPECIALのオンラインマッチメイクを悪用するチート行為で，主に日本サーバーで確認されています．名前の由来は，犯行者のゲーム内タグが実在のトッププレイヤーの名前を無断使用していることによるものです (実際の本人は無関係であり，界隈全体で周知されています)．

### 仕組み

クイックプレイ・VIPマッチで対戦相手を検索すると，NEXマッチメイクサーバーによりセッションに割り当てられ，通常の1v1では`max_participants=2`となりますが，チーターはCFW (カスタムファームウェア) を使い，このプロセスを操作しています:

**4/4型**: 1v1にもかかわらず`max_participants=4`のセッションを作成し，複数の一般プレイヤーが同一セッションに引き込まれ，ゲーム側が「満員」と判断しエラーを出します．全被害者の切断回数カウントが増えます．

**2/2型**: セッション情報は正常 (`max_participants=2`) ですが，PIA接続フェーズで何らかの方法によりエラーを誘発します．被害者の切断回数カウントが増えます．

いずれの場合も:
- NEXサーバーはエラーを返さない (異常を検知していない)
- Switchの「いっしょにあそんだ人」にはチーターのアカウント1つと，巻き込まれた一般プレイヤー2-3人が表示される
- チーターは大量のアカウントを使用している (CFWによる偽装の可能性)

### 問題点

- **Switchのブロック機能では対処できない** — ブロック枠の上限をすべて消費しても，次々と別のアカウントでマッチングしてくる
- **被害者だけがペナルティを受ける** — レートの上下は双方発生しないが，エラーの原因はチーター側にあるにもかかわらず，被害者のみが切断回数カウントを受け取る．累積するとマッチメイク制限などのシステムペナルティにつながる
- **レートが高いほど遭遇率が上がる** — チーターはふだん，エラー流しとは別に，ラグスイッチや体力制 + チャージ切り札のゲーム＆ウォッチなどあちら側が一方的に有利となるルール設定でレートを盛っている場合が多いため，上位帯では連続で引き当てることが日常茶飯事であり，1時間で数試合しかできないなどまともに遊べない状況が発生する
- **任天堂側に検知手段がない** — NEXサーバーは全レスポンスを正常に返しており，サーバー側からはチートが見えていない

### 解明できたこと

| 発見事項 | 詳細 |
|----------|------|
| 4/4型は検知可能 | 1v1セッションで`max_participants > 2` |
| NEXはエラーを出さない | 全RMCレスポンスが正常，サーバーは異常を認識していない |
| 攻撃ごとに複数の被害者 | NAT traversal (proto=14 method=1) が通常の2回ではなく4回以上発火 |
| PIAパケットドロップでペナルティ回避可能 | suspectセッションのPIAパケットを破棄することで切断カウントが加算されない |
| `app_buf[0]`で判別可能 | `0x01`=自己ホスト (正常)，`0x02`=2/2型エラー，`0x03`/`0x04`/`0x06`=4/4型エラー，`0x05`/`0x09`=正常 |
| プレイヤー名が取得可能 | `app_buf+0x1e`にUTF-16LEでセッションオーナーの名前が格納されている |
